# Scientific reference data

This directory contains version-controlled input and expected-output values used
as scientific test oracles by the FRsutils test suite. The values support
regression tests for fuzzy-rough operators, fuzzy quantifiers, similarity
computations, OWA weights, and approximation models.

The JSON files are intentionally kept human-readable so reviewers can inspect
numerical changes in pull-request diffs. `manifest.json` records the expected
SHA-256 digest and dataset kind for each reference-data file. The shared loader
in `tests/reference_data_loader.py` verifies those digests, rejects ambiguous or
non-standard JSON, restores NumPy arrays with their declared dtype and shape,
and marks the arrays as read-only.

The migration of canonical scientific oracles from Python literals to these
JSON files was completed on 2026-07-13. The one-off extraction script and
migration inventory were removed after validation; Git history preserves the
migration record. Local fixtures that test validation, shape, dtype, routing,
or edge-case behavior remain in their test modules.

## Reference-data change policy

A failing implementation test is not, by itself, a reason to update an expected
value. Before changing reference data, determine whether the failure comes from:

1. an implementation defect,
2. an incorrect historical reference value,
3. a deliberately changed mathematical convention, or
4. an intentionally changed public or scientific contract.

Reference values must not be regenerated by calling the production function
that the same test is intended to validate. Acceptable derivations include:

- a hand calculation for a small, inspectable case,
- direct evaluation of the defining mathematical formula,
- an independent and deliberately simple reference implementation,
- a worked example from a cited scientific source, or
- agreement with a separately validated external implementation, with its
  version and convention documented.

Every reference-data change must include:

- a concise mathematical or scientific justification,
- the independent derivation or verification procedure,
- updated provenance metadata in the affected JSON file,
- the updated SHA-256 digest in `manifest.json`,
- tests covering the intended behavior or corrected convention, and
- explicit approval from the code owner.

Keep reference-data changes separate from production-code changes whenever
possible. Prefer a dedicated pull request; otherwise, use a dedicated commit so
the numerical diff and its justification remain easy to review.

Do not weaken a test by increasing its tolerance merely to accommodate a changed
expected value. Numerical tolerances belong in test code and require their own
scientific justification.

## File format

Each data file has this top-level structure:

```json
{
  "schema_version": 1,
  "dataset_kind": "example_kind",
  "provenance": {
    "origin": "description of the independent source or migration source",
    "verification": "description of how the values were checked"
  },
  "data": []
}
```

The `data` field may be a list of homogeneous cases or an object that groups
multiple related case collections while preserving compatibility accessors.
Dataset-specific contract tests validate the grouped schemas used by implicator,
similarity, approximation-baseline, and fuzzy-rough model reference data.

NumPy arrays use an explicit representation:

```json
{
  "__ndarray__": [0.0, 0.5, 1.0],
  "dtype": "float64",
  "shape": [3]
}
```

Keep the declared dtype and shape exact. The loader currently supports
`float64` and `int64` reference arrays and rejects `NaN`, `Infinity`, duplicate
keys, and ragged shapes. The contract tests also reject JSON files that are not
registered in the manifest.

The model files keep historical parameterized cases under `legacy_testsets`.
`itfrs.json` and `owafrs.json` store focused dense regression oracles under
`dense_baselines`. `vqrs.json` stores each shared matrix-and-label fixture once
under `dense_fixtures`, with one or more component configurations and expected
outputs in its `baselines` collection. The compatibility accessors in
`tests/reference_data_store.py` expose a flat list of dense VQRS cases so tests
remain isolated from this storage-level deduplication.

String or mixed-type labels used by dense VQRS and OWAFRS baselines are stored
as plain JSON values with explicit metadata rather than encoded NumPy arrays:

```json
{
  "values": ["a", "a", "b", "b"],
  "dtype": "object",
  "shape": [4]
}
```

This avoids extending the numeric array decoder to object arrays while still
making label type and shape changes reviewable. Tests reconstruct these labels
as local NumPy arrays when required.

## Updating a file safely

1. Re-derive or independently verify the affected values.
2. Edit only the necessary JSON entries and update their provenance.
3. Compute the new digest:

   ```bash
   python -c "from pathlib import Path; import hashlib; p=Path('tests/reference_data/FILE.json'); print(hashlib.sha256(p.read_bytes()).hexdigest())"
   ```

4. Replace only that file's `sha256` value in `manifest.json`.
5. Run the reference-data contract tests:

   ```bash
   python -m pytest tests/reference_data_tests -ra
   ```

6. Run every scientific test module that consumes the changed dataset.
7. Explain the derivation and affected behavior in the pull request.

Never update all hashes automatically without first inspecting the JSON diff.
A digest confirms that reviewed bytes have not changed; it does not establish
that the scientific values are correct.

## Repository protection

`/.github/CODEOWNERS` assigns this directory, its loader, compatibility accessors,
and contract tests to the project maintainer. On the protected default branch,
repository settings should require pull requests, successful status checks, and
approval from code owners. CODEOWNERS identifies required reviewers but does not
block direct changes unless branch protection or rulesets enforce that policy.
